# レビュー出力フォーマット

本リポジトリの語彙を用いて指摘・提案を行うときの出力テンプレート

将来のレビュー Skill が遵守する仕様であり、人間がレビューコメントを書くときの参考にもなる

## 共通原則

`vocab-*.md` の各エントリは「定義 / 使うとき / 適用すべきケース / 適用すべきでないケース / トレードオフ / コード例 / 関連」の構造を持つ  これをそのままレビュー指摘の構造として再利用する

レビュー前のチェック手順:

1. 対象コード / 状況から [INDEX.md](INDEX.md) で症状を特定
2. 候補語彙の `vocab-*.md` を読む
3. **「適用すべきでないケース」と必ず照合** → 該当する場合は指摘しない (誤検出フィルタ)
4. 残った指摘を本ファイルのテンプレートで出力

## 個別指摘テンプレート

各指摘は以下の 5 要素で構成する  4 点セット表現 ([WRITING-RULES.md F1〜F8](WRITING-RULES.md)) を必ず含める

```md
### [優先度] {語彙名} — {対象 (ファイル名:行 / クラス / メソッド)}

**4 点セット**: **{語彙名}** の観点から、**{現状 X}** を **{提案 Y}** にすると、**{改善される性質 Z}** が改善される

**トレードオフ**: {犠牲になる性質} ↔ {得られる性質}  → {判断軸}

**今すぐ直すか / 後でいいか**: {即時 / Boy Scout Rule / 別 PR / Tech Debt 化}  → {理由}

**参照**: [{vocab-X.md の該当エントリ}](vocab-X.md#{anchor})
```

**優先度** は次の 3 段階:

- **🔴 高**: 公開 API / DB スキーマ / 業務クリティカルなど、後から直すコストが大きい箇所
- **🟡 中**: 動作は問題ないが保守性・テスト容易性で明確に劣化が出る箇所
- **🟢 低**: Boy Scout Rule で次回ついでに直せばよい箇所

## 全体出力テンプレート

複数指摘を束ねるときは以下の構造で出力する

```md
# レビュー結果 — {対象範囲 (PR タイトル / ファイル名 / コミット範囲)}

## TL;DR

- 検出した指摘: {🔴 高: N 件 / 🟡 中: N 件 / 🟢 低: N 件}
- 推奨アクション: {即時対応 / 別 PR / そのまま進める / etc.}
- 主要な要注意箇所: {1-3 件、語彙名で簡潔に}

## 指摘一覧

{個別指摘テンプレートを優先度順に列挙}

## 検出したが指摘しなかったもの (誤検出排除のログ)

| 候補語彙 | 対象 | 適用すべきでない理由 |
|---|---|---|
| {語彙名} | {対象} | {「適用すべきでないケース」のどれに該当したか} |

## 全体トレードオフ

{個別指摘を統合した上での全体的な判断軸  例: 「この PR は試作期なので SRP/DIP の厳密適用より速度を優先」}
```

## 出力例 (架空のレビュー)

```md
# レビュー結果 — feat/order-place

## TL;DR

- 検出した指摘: 🔴 高: 1 / 🟡 中: 2 / 🟢 低: 1
- 推奨アクション: 🔴 のみ本 PR で対応、🟡 は別 PR、🟢 は次回 Boy Scout で
- 主要な要注意箇所: Long Method (OrderController::place), Hard-coded Dependency (Mailer)

## 指摘一覧

### 🔴 高 Long Method — `app/Http/Controllers/OrderController.php:42-118`

**4 点セット**: **Long Method** の観点から、**バリデーション・料金計算・永続化・通知が同居している 76 行のメソッド** を **Extract Method で 4 つの目的別 private メソッドに分割** すると、**メソッド名から意図が読める可読性とユニットテスト容易性** が改善される

**トレードオフ**: 短さ ↔ 過剰分割で全体像が見えない  → 「1 メソッド = 1 目的 + 1 抽象レベル」が基準

**今すぐ直すか / 後でいいか**: 即時 (本 PR で対応)  → このメソッドが本 PR の主役で、追加レビュアーの読解負荷に直結

**参照**: [vocab-code-quality-refactoring.md の Long Method](vocab-code-quality-refactoring.md)

### 🟡 中 Hard-coded Dependency — `app/Services/OrderService.php:30`

**4 点セット**: **Hard-coded Dependency** の観点から、**`new SmtpMailer()` を内部で直接生成している現状** を **Constructor Injection** にすると、**テスト時の Fake 差し替え可能性とユニットテストの実行速度** が改善される

**トレードオフ**: 単純さ ↔ テスト容易性  → 外部 I/O 依存は DI 推奨

**今すぐ直すか / 後でいいか**: 別 PR  → DI 導入は本 PR のスコープ外、別 PR で OrderService 全体の依存整理とセットで行う方が良い

**参照**: [vocab-testability.md の Hard-coded Dependency](vocab-testability.md)

## 検出したが指摘しなかったもの

| 候補語彙 | 対象 | 適用すべきでない理由 |
|---|---|---|
| SRP | OrderController | この Controller は HTTP 境界のみ担当で、業務判断は UseCase に切り出されている  SRP 違反に該当せず |
| DIP | OrderController が OrderRepository に依存 | interface を介して注入されており、DIP は既に満たされている |

## 全体トレードオフ

本 PR は新規機能の最初の実装で、要件が固まる前段階  
Long Method (🔴) はレビュー読解直結なので即時対応、Hard-coded Dependency (🟡) は別 PR で OrderService 全体の依存整理と合わせて実施するのが妥当
```

## 指摘を出さない判断基準 (重要)

公式ベストプラクティスの「Style guide validator」パターンに従い、**指摘を出すかどうか** は次の質問でセルフチェックする:

1. **適用すべきでないケースに該当しないか** ([WRITING-RULES.md 共通ルール 4](WRITING-RULES.md))
2. **指摘の {改善される性質 Z} が測れる / 語れる性質か** (「綺麗になる」「気持ちいい」は NG、[WRITING-RULES.md 共通ルール 2](WRITING-RULES.md))
3. **4 点セット表現で言語化できるか** (語彙名だけ振りかざしてないか、[共通ルール 1](WRITING-RULES.md))
4. **トレードオフを併記できるか** (指摘だけで止めてないか、[共通ルール 5](WRITING-RULES.md))
5. **「今すぐ / 後で」の判断軸まで言えるか** ([共通ルール 5](WRITING-RULES.md))

5 つすべて Yes でなければ指摘を出さない  Yes にできなければ「気になるが指摘しない」セクションに移して、誤検出排除ログとして可視化する

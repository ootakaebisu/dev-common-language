# 判断軸・トレードオフ・メタ語彙

「もっと綺麗にしたい」「将来◯◯になるかも…」を **「{判断軸} の観点から {コスト/リスク} と {便益} を比較すると {結論}」** 表現に変換するための語彙
原則を当てずっぽうに振り回す前に「**今この場面でその原則を出すのは妥当か**」を判断するためのメタ語彙  **ノイズ排除の本丸**


## TL;DR
- 「**判断軸**」は議論のレベルを 1 段上げる語彙  原則 (どう書くか) ではなく **メタ (今書くべきか)** を扱う
- このカテゴリを身につけると、相手の判断ロジックを**先回りで読める**ようになる  これが「フラットな議論」の本質
- **「Two-way door / One-way door」** と **「Premature Optimization」** と **「Bikeshedding」** の 3 つだけ覚えてもノイズ発言は劇的に減る

## 過剰設計を止める判断軸

### Premature Optimization
- **読み**: プリマチュア オプティマイゼーション
- **定義**: 早すぎる最適化  ボトルネックの計測なしに性能改善を試みること  Knuth 「Premature optimization is the root of all evil」
- **使うとき**:
  - **Premature Optimization** の懸念がある。**計測なしに最適化判断** すると、可読性を犠牲にした割に効果が小さい可能性がある。**計測してから判断** 検討すべき。
- **止めるノイズ発言**: 「ここキャッシュした方が速いと思います」「並列化したらどうですか」「N+1 直する」(← 計測なしで言うと刺さる)
- **適用すべきでないケース**:
  - **既に計測済みでホットスポット確定**してる場合は最適化推進が正
  - **明らかな O(n²) の入れ子ループ**等、計測前提でなく改善できる箇所
- **関連**: YAGNI / Pareto / 計測駆動

### YAGNI
- **読み**: ヤグニ (You Aren't Gonna Need It)
- **定義**: 今いらない機能・抽象は作らない  必要になったときに作る
- **使うとき**:
  - **YAGNI** の観点から、{将来要件 Z} のための拡張ポイントは、**実要件発生時に追加** する方が、**現スコープの実装/レビュー/テスト総コスト** が小さくなる。
- **止めるノイズ発言**: 「将来◯◯になるかもなので拡張点を…」「いつか使うかもしれないので interface を…」
- **適用すべきでないケース**:
  - **後付けが極めて高コスト**な箇所 (DB スキーマ、公開 API 契約、データ移行を伴う構造)
  - セキュリティ・監査・法令対応で高確率で必要が確定してる箇所
- **トレードオフ**: 今のコスト ↔ 後付けコストの非対称性
- **関連**: KISS / Speculative Generality / Last Responsible Moment

### Speculative Generality
- **読み**: スペキュレイティブ ジェネラリティ
- **定義**: 投機的一般化  「いつか使うかも」で作った抽象が使われずに残ること
- **使うとき**:
  - **Speculative Generality** の懸念がある。使われてない抽象を残すと、**読み手の認知負荷とメンテ対象範囲** が増える。
- **関連**: YAGNI / KISS

### Bikeshedding
- **読み**: バイクシェディング
- **定義**: 些末な議題 (色・名前) ほど誰もが意見を持ち、議論時間を溶かす現象
- **使うとき**:
  - これは **Bikeshedding** になりかけている。**本質的論点 ({重要論点 X})** に時間を使った方が良さそう。
- **止めるノイズ発言**: 命名の細部議論を 30 分続ける / インデント論争 / 些末な型名選定
- **適用すべきでないケース**:
  - **公開 API の命名**は些末ではない (一度公開すると変更コスト大)
  - Ubiquitous Language の命名 (業務側との接続点)
- **関連**: Last Responsible Moment

## 可逆性・タイミング判断

### Two-way door / One-way door
- **読み**: ツーウェイ ドア / ワンウェイ ドア
- **定義**: **可逆な判断 (Two-way door)** は早く決めて試す  **不可逆な判断 (One-way door)** は時間をかけて慎重に
- **使うとき**:
  - これは **Two-way door** なので、議論で決めずに **小さく試して測る** 方が早い。
  - これは **One-way door** (DB スキーマ / 公開 API / データ移行) なので、**もう一段慎重に検討** する価値がある。
- **適用すべきケース**:
  - 議論が長引いてる時
  - 決定の可逆性を可視化したい時
- **判断基準**:
  - Two-way: 1 PR で戻せる / 内部実装 / 機能フラグで制御可能
  - One-way: DB スキーマ / 公開 API / データ移行 / 外部契約
- **関連**: Last Responsible Moment / Cost of Change

### Last Responsible Moment / LRM
- **読み**: ラスト レスポンシブル モーメント / エルアールエム
- **定義**: 決定を **責任を持って遅らせられる最後の瞬間** まで遅らせる (Lean Software Development)
- **使うとき**:
  - **Last Responsible Moment** の観点から、{この判断} は **今決めなくても先に進める** ので、**もう少し情報が出てから決める** 方が良さそう。
- **適用すべきケース**:
  - 情報が足りない状況
  - 試作期
  - 要件が固まってない
- **適用すべきでないケース**:
  - **判断を遅らせると詰む箇所** (DB スキーマ確定がブロッカー等)
  - 単なる**先送り**にしてしまう (責任を持つ意思が必要)
- **関連**: YAGNI / Two-way door

### Cost of Change Curve
- **読み**: コスト オブ チェンジ カーブ
- **定義**: 変更コストはフェーズが進むほど指数的に増える (旧説) / アジャイル + 良い設計でフラット化できる (新説)
- **使うとき**:
  - **Cost of Change** の観点から、この **One-way door** は本番デプロイ後の変更コストが極端に高くなるので、今のフェーズで決めるのが妥当。
- **関連**: Two-way door / Tech Debt

## 優先順位判断

### Pareto / 80-20
- **読み**: パレート
- **定義**: 80% の効果は 20% の箇所から生まれる  改善は集中的に
- **使うとき**:
  - **80-20** の観点から、**ホットスポット ({ファイル X})** だけ先に改善する方が ROI が高い。全体一斉リファクタは避けたい。
- **止めるノイズ発言**: 「全部リファクタしたい」「全機能テスト追加したい」
- **関連**: Premature Optimization

### Make it work → Make it right → Make it fast
- **読み**: メイク イット ワーク → ライト → ファスト
- **定義**: まず動かせ → 次に正しく → 最後に速く  の順を守る
- **使うとき**:
  - **Make it work が先**なので、**動く状態を最短で作って** から、Make it right (リファクタ) と Make it fast (最適化) は後続でやりましょう。
- **止めるノイズ発言**: 「最初から綺麗に書きたい」「先に最適化した方が…」
- **適用すべきでないケース**:
  - **本番投入後**にこの順を持ち出してリファクタを永遠に先送り
- **関連**: Premature Optimization / YAGNI

### Hot Path / Cold Path
- **読み**: ホット パス / コールド パス
- **定義**: 頻繁に通る経路 (Hot) と滅多に通らない経路 (Cold)  最適化と複雑性投資は Hot に集中
- **使うとき**:
  - これは **Cold Path** なので、**シンプルさを優先** して最適化は不要。

### Tech Debt
- **読み**: テック デット
- **定義**: 技術的負債  短期速度のために選んだ次善策が、長期コストとして利息を生む (Ward Cunningham のメタファ)
- **使うとき**:
  - これは **Tech Debt として登録** して、{条件 X} を満たしたら返済する方針が妥当。
- **適用すべきケース**: 短期に動かす必要があり、後で返済可能な箇所
- **適用すべきでないケース**:
  - **負債登録だけして永久に返さない**運用 (= ただの言い訳)
  - **負債と呼ぶべきでない設計上の正解**を負債扱いして混乱させる
- **関連**: Last Responsible Moment / Boy Scout Rule

### Boy Scout Rule
- **読み**: ボーイスカウト ルール
- **定義**: 「コードベースをチェックアウトした時より少しでも綺麗にしてコミットする」
- **使うとき**:
  - ここは **Boy Scout Rule** で、この PR のついでに小さく直しておく。別 PR 化するほどでもない。
- **適用すべきでないケース**:
  - **PR 肥大化**してレビュー困難になる規模の修正を「Boy Scout で」と紛れ込ませる
- **関連**: Tech Debt / レビュー粒度

## 議論の構造化

### Stop the bleeding
- **読み**: ストップ ザ ブリーディング
- **定義**: 出血を止める  インシデント時、まず影響拡大を止める  根本原因究明と恒久対応は次のステップ
- **使うとき**:
  - 今は **Stop the bleeding** を優先して、{応急処置} で動かしてから、**根本原因と恒久対応は別チケットで** やりましょう。
- **関連**: Tech Debt / インシデント対応

### Reversible / Irreversible
- **読み**: リバーシブル / イレバーシブル
- **定義**: 戻せる判断か戻せない判断か  失敗コストの見積もり語彙
- **使うとき**:
  - これは **Reversible** なので、**まず試して測る** 方が議論より早い。
- **関連**: Two-way door

### Conway's Law
- **読み**: コンウェイズ ロー
- **定義**: システムの構造は、それを設計する組織のコミュニケーション構造を反映する
- **使うとき**:
  - **Conway's Law** の観点から、この境界は **チーム境界と一致してる方が**長期的にメンテしやすい。
- **適用すべきケース**:
  - チーム横断機能
  - Bounded Context の境界設計
- **関連**: Bounded Context / Team Topologies

### Build vs Buy
- **読み**: ビルド バーサス バイ
- **定義**: 自作するか、外部サービス/ライブラリを使うか の判断
- **使うとき**:
  - **Build vs Buy** で、自作の **学習コスト + メンテコスト** と Buy の **ライセンス + ロックイン** を比較検討すべき。

### Throwaway code vs Production code
- **読み**: スローアウェイ コード / プロダクション コード
- **定義**: 使い捨てコードと本番コードでは、品質基準と投資量を変える
- **使うとき**:
  - これは **Throwaway** なので、リファクタやテスト追加は不要。Production 化が決まってから設計する。
- **適用すべきでないケース**:
  - 「Throwaway」と言って書いたコードが**気づいたら本番で動いてる**パターン (要警戒)

## 認知のメタ概念

### Confirmation Bias
- **読み**: コンファメーション バイアス
- **定義**: 確証バイアス  自分の仮説を支持する情報ばかり集めてしまう傾向
- **使うとき**:
  - **Confirmation Bias** に注意して、{反対の事例 / 反例データ} も見る選択肢を検討する。

### Cargo Cult Programming
- **読み**: カーゴ カルト プログラミング
- **定義**: なぜそうするかを理解せず、形だけ真似る  「{有名 OSS} もこうしてたから」だけで採用
- **使うとき**:
  - これは **Cargo Cult** にならないよう、**なぜそうするか** を一度確認検討すべき。

### Hyrum's Law
- **読み**: ハイラムズ ロー
- **定義**: 「API の利用者が十分にいれば、APIの仕様 (ドキュメントに書かれた契約) に関係なく、観測可能な振る舞い全てが誰かに依存される」
- **使うとき**:
  - **Hyrum's Law** の懸念があるので、内部実装の変更でも **利用者の振る舞いに影響しないか** 確認検討すべき。
- **適用すべきケース**:
  - 公開 API
  - SDK
  - よく使われる内部ライブラリ

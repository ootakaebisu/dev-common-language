# コミュニケーション (UI / 文書 / 会話)

「伝わらない」「読みにくい」「離脱される」を **「{原則} の観点から {現状 X} を {提案 Y} にすると {読み手の負荷 Z} が下がる」** 表現に変換するための語彙
ワーキングメモリの限界 (4±1 チャンク) を前提に、**UI / 文書 / 会話・プレゼン** で読み手 / 聞き手の理解コストを下げる共通言語

コード領域の認知負荷は [vocab-cognitive-load.md](vocab-cognitive-load.md) を参照

## TL;DR
- 「伝わらない」を語る前に **「想定読み手は誰か」** を明示する  読者層が違えば適切な情報密度も違う
- 同じ原則 (Chunking / Schema / Selective Emphasis) がコード領域でも効く ([vocab-cognitive-load.md](vocab-cognitive-load.md))
- 強調は **「絞ってこそ機能する」**  全部目立たせると何も目立たない (色 / 太字 / 大文字 すべて共通)
- 結論ファースト (TL;DR / BLUF / Pyramid) は **読まれる確率と理解速度を同時に上げる** 唯一の手段

## 視覚情報・UI

### Hick's Law
- **読み**: ヒックス ロー
- **定義**: ヒックの法則  選択肢の数が増えると意思決定時間が対数的に増える (T = a + b log₂(n+1))
- **使うとき**: **Hick's Law** の観点から、{1 画面 12 ボタン} を **主操作 3 + その他をメニュー** に分けると **選択コスト** が下がる。
- **適用すべきケース**:
  - 主導線設計
  - ナビゲーション
  - フォーム
- **適用すべきでないケース**: 専門ツール (Photoshop 等) で**意図的に全機能露出**するケース
- **トレードオフ**: 階層化 ↔ 目的の機能への到達クリック数
- **関連**: Working Memory / Selective Emphasis

### Fitts's Law
- **読み**: フィッツイス ロー
- **定義**: フィッツの法則  ターゲットへの到達時間はターゲットまでの距離に比例し、ターゲットの大きさに反比例する  「重要操作は大きく / 近くに / 端に」
- **使うとき**: **Fitts's Law** の観点から、**主要 CTA を画面端 + サイズ大** にすると **物理的到達コスト** が下がる。
- **適用すべきケース**:
  - ボタン配置
  - モバイル UI
  - 頻度高い操作
- **適用すべきでないケース**: 破壊的操作 (削除等) を「使いやすく」してしまう
- **トレードオフ**: 操作しやすさ ↔ 誤操作リスク
- **関連**: Selective Emphasis

### Gestalt Principles
- **読み**: ゲシュタルト プリンシプルズ
- **定義**: ゲシュタルト原則  近接・類同・閉合・連続・対称 等の知覚原理  視覚的にまとまって見える要素は「同じ」と認識される
- **使うとき**: **Gestalt の近接原則** で、関連項目を**近く + グループ間に余白** にすると **論理構造の認識コスト** が下がる。
- **適用すべきケース**:
  - 一覧表示
  - フォームのグループ化
  - カード UI
- **適用すべきでないケース**: 物理的に近いだけで本来関連のない要素をまとめると誤認を生む
- **トレードオフ**: グルーピングの強さ ↔ 個別要素の独立性
- **関連**: Whitespace / Chunking

### Whitespace
- **読み**: ホワイトスペース
- **定義**: 余白 / グルーピング  要素間の空白  情報密度を下げ、グルーピングを示し、視線誘導する
- **使うとき**: **Whitespace** を増やして要素間グループを明示すると **論理構造の同時把握コスト** が下がる。
- **適用すべきケース**:
  - 密集レイアウト
  - 階層構造が見えにくい画面
- **適用すべきでないケース**: 1 画面で全情報を見せる必要があるダッシュボード (スクロール代償)
- **トレードオフ**: 情報密度 ↔ 視認性
- **関連**: Gestalt Principles / Information Density

### Color Palette Limit
- **読み**: カラー パレット リミット
- **定義**: 色数の制限  1 画面の色数は意味付きで 5±2 程度  「全部目立つ = 何も目立たない」
- **使うとき**: 色を **意味のある 3〜5 色** に制限すると **意味の区別認識コスト** が下がる。
- **適用すべきケース**:
  - ダッシュボード
  - アラート設計
  - グラフ
- **適用すべきでないケース**: 装飾用途 (マーケLP) の意図的な多色配色
- **トレードオフ**: 表現力 ↔ 認識コスト
- **関連**: Selective Emphasis

## 文書・ドキュメント

### TL;DR
- **読み**: ティーエル ドクター
- **定義**: 要約先出し / Summary First  結論・要旨を文書冒頭に置く  読者が先に「全体像」をワーキングメモリに乗せてから詳細を読める
- **使うとき**: **TL;DR** を冒頭に追加すると **読み手が詳細をワーキングメモリに保持する必要が減る**。
- **適用すべきケース**:
  - PR 説明
  - 設計書
  - 報告書
  - ブログ記事
- **適用すべきでないケース**: ミステリー的に結論を最後に置きたい文章 (小説等)
- **トレードオフ**: 構成の自由度 ↔ 読み手効率
- **関連**: BLUF / Pyramid Principle

### Heading Hierarchy
- **読み**: ヘディング ヒエラルキー
- **定義**: 見出し階層  H1/H2/H3 の階層で文書構造を示す  目次として ワーキングメモリ上に「地図」を作る
- **使うとき**: **Heading Hierarchy** を 3 段以内 + 各見出し短文 にすると **読み手の現在位置把握コスト** が下がる。
- **適用すべきケース**:
  - 長文ドキュメント
  - Markdown 文書
  - 仕様書
- **適用すべきでないケース**: 1 トピック完結の短文に見出しを無理に付ける
- **トレードオフ**: 構造化 ↔ 過剰な分割
- **関連**: TL;DR / Schema

### One Idea per Paragraph
- **読み**: ワン アイディア パー パラグラフ
- **定義**: 段落 1 主張  1 段落 = 1 つの主張  複数主張が混ざるとワーキングメモリで分離処理が必要になる
- **使うとき**: 段落が長い。**1 段落 1 主張** で分割すると **論理の追跡コスト** が下がる。
- **適用すべきケース**:
  - 説明文
  - 議事録
  - レビュー文章
- **適用すべきでないケース**: 詩的表現や物語のように複数要素を意図的に絡める文章
- **トレードオフ**: 分割の明快さ ↔ 段落数の増加
- **関連**: SRP (文書版) / TL;DR

### Term Consistency
- **読み**: ターム コンシステンシー
- **定義**: 用語統一  同じ概念は同じ言葉で呼ぶ  「ユーザー / 利用者 / メンバー」を混在させると「これは同じものか？」の判定コストが発生
- **使うとき**: **Term Consistency** の観点から、{ユーザー / 利用者} を **「ユーザー」に統一** すると **同一性判定コスト** が下がる。
- **適用すべきケース**:
  - 設計書
  - API 仕様
  - UI 文言
- **適用すべきでないケース**: 文脈で意味が違う場合に無理に統一 (= 別概念が同名化)
- **トレードオフ**: 統一 ↔ 表現の豊かさ
- **関連**: Ubiquitous Language / Consistency

### Visual Aid
- **読み**: ビジュアル エイド
- **定義**: 図解  図/表/フロー  文字列で長文化するより、空間配置で構造を示す方がワーキングメモリの効率が良い
- **使うとき**: {文章で 5 段の階層関係を説明} を **木構造の図** にすると **構造把握コスト** が下がる。
- **適用すべきケース**:
  - 階層構造
  - フロー
  - 関係性
  - 比較表
- **適用すべきでないケース**: 単純な内容に図を作って **図解のメンテコスト** だけ残る
- **トレードオフ**: 作成・メンテコスト ↔ 読み手の理解効率
- **関連**: Schema / Gestalt Principles

## 会話・プレゼン

### PREP
- **読み**: プレップ
- **定義**: Point (主張) → Reason (理由) → Example (例) → Point (再確認) の構造  口頭/短文の汎用テンプレ
- **使うとき**: **PREP** で組み立てると、聞き手が **主張を先にワーキングメモリの確保** してから根拠を受け取れて理解負荷が下がる。
- **適用すべきケース**:
  - 朝会発言
  - 短いプレゼン
  - 短文回答
- **適用すべきでないケース**: 長時間の物語的プレゼン (構造が単調になる)
- **トレードオフ**: 構造の明快さ ↔ 表現の自由度
- **関連**: BLUF / Rule of Three

### Bottom Line Up Front
- **読み**: バ゙ム
- **定義**: 結論ファースト  軍/外交由来  まず結論、次に背景  長文メールや報告書で「読まれる確率」を上げる
- **使うとき**: **BLUF** で冒頭に結論を置くと **読み手が背景情報をワーキングメモリに保持する負担** が下がる。
- **適用すべきケース**:
  - メール
  - Slack 長文
  - 報告書
- **適用すべきでないケース**: 説得が必要で結論先出しすると拒絶される文脈 (営業 / クレーム対応)
- **トレードオフ**: 効率 ↔ 説得力
- **関連**: TL;DR / Pyramid Principle

### Pyramid Principle
- **読み**: ピラミッド プリンシプル
- **定義**: のピラミッド原則  1 つの結論 → 3 つの根拠 → 各根拠の詳細  上から下に降りていく構造で全体像→詳細の順でワーキングメモリに積む
- **使うとき**: **Pyramid Principle** で、結論 1 → 根拠 3 → 詳細 にすると **論理追跡のワーキングメモリのコスト** が下がる。
- **適用すべきケース**:
  - コンサル提案
  - 経営報告
  - 設計レビュー資料
- **適用すべきでないケース**: 探求型の議論 (結論が固まっていない発散フェーズ)
- **トレードオフ**: 構造の硬さ ↔ 探求の柔軟性
- **関連**: Rule of Three / BLUF

### Rule of Three
- **読み**: ルール オブ スリー
- **定義**: つの法則  人は 3 つを最も自然に把握できる  根拠 / 例 / ポイントは 3 つに絞るとワーキングメモリの効率が最大
- **使うとき**: {論点 7 個} を **重要 3 個に絞る** と **聞き手のワーキングメモリの占有** が下がる。
- **適用すべきケース**:
  - プレゼン構成
  - FAQ
  - 提案ポイント
- **適用すべきでないケース**: 網羅性が要件 (法務 / 監査 / チェックリスト)
- **トレードオフ**: 絞り込みの明快さ ↔ 網羅性
- **関連**: Hick's Law / Working Memory

## トレードオフ・判断軸

### Selective Emphasis
- **読み**: セレクティブ エンファシス
- **定義**: 絞った強調  「全部目立たせる = 何も目立たない」  本当に重要なものだけ強調する
- **使うとき**: **Selective Emphasis** で、強調は **1 画面 / 1 段落で最大 1〜2 箇所** に絞ると **重要箇所の検出コスト** が下がる。
- **判断基準**: 強調要素が全体の 20% を超えたら警戒
- **関連**: Color Palette Limit / Pareto

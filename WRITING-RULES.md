# 記述ルール

語彙集を書く・更新する・フィードバックするときに従う共通ルール

各 `vocab-*.md` ファイルに個別の注意を書かず、本ドキュメントに集約する

## 共通ルール (どのカテゴリでも守る)

### 1. 4 点セットで言う

**「{概念} の観点から {現状 X} を {提案 Y} にすると {改善される性質 Z} が改善される」**

- 概念名だけ振りかざすのは水掛け論 (例:「SRP 違反」「Long Method」「依存が逆」だけで止めない)
- {改善される性質 Z} は **測れる / 語れる** 性質を選ぶ (変更容易性 / テスト容易性 / 可読性 / 凝集度 / 結合度 等)
- 「臭い → リファクタリング手法 → 改善される性質」のような派生形も同じ構造

### 2. 抽象語彙は禁止

「気持ち悪い」「汚い」「綺麗にしたい」「読みづらい」は禁止語彙  必ず具体化する

- 「綺麗になる」「気持ちいい」は性質 Z にカウントしない
- 「私には読みづらい」と「多くの読み手にとって読みづらい」を分ける  自分の知識ギャップを問題化しない

### 3. 原典マウント禁止

「Clean Architecture では…」「DDD だと…」「リファクタリング本では…」と教科書を振りかざさない

- 起点は **現コードベースで合意済みの構成 / 業務ドメインの Ubiquitous Language**
- ゼロベース議論は別の場で

### 4. 「適用すべきでないケース」を必ず確認

原則・概念は **万能カード化しやすい**  提案/反論する前に「ここでは適用しない方がいいケース」を確認する

- 各エントリの「適用すべきでないケース」欄を最優先で読む
- 万能カードに見えたら、それは抽象化の解像度が足りてないサイン

### 5. コストと便益はセットで提示

語彙は**振りかざせばマウント凶器**になる  指摘で止めない

- 「YAGNI ですよ」「SRP 違反です」「臭いがあります」だけで終わらず、**「だから今は◯◯にしよう」**まで言う
- 「今すぐ直すか / 後でいいか」もセット (Boy Scout Rule / 別 PR / Tech Debt 化)
- 判断軸は **自分の主張を補強する道具にも、相手を黙らせる凶器にもなる**  自戒する

### 6. 依存・対象を明示する

抽象な指摘は曖昧さでマウントになる  対象を具体化する

- 「依存方向が逆」→ **何から何への依存** かを必ず明示
- 「テストしやすくしよう」→ **どの種類のテストの何が改善されるか** を明示 (ユニット実行速度 / モック容易性 / E2E 安定性 等)
- 「認知負荷が高い」→ **どのチャンクが何個増えてるか** まで具体化する

## フォーマットルール (記述形式)

各 `vocab-*.md` ファイルのエントリを書くときの形式ルール  整備のたびに本セクションに追記し、再現性を担保する

### F1. 略語は使わず正式名で書く

項目単体で読んだとき意味が通るよう、本文中で略語を使わない  略称が項目タイトル (SRP / OCP / DI / ACL / DTO 等) になっているものは、その項目内では略称を使ってよい  それ以外の領域語彙の短縮は禁止

| 禁止 | 推奨 |
|---|---|
| WM | ワーキングメモリ |
| VO | Value Object |
| FW | フレームワーク |
| IF (interface の意味) | インターフェース |

カタカナ語に助詞が続くときはスペースを入れず連結 (例: 「ワーキングメモリを超える」)  名詞修飾には「の」を挟む (例: 「ワーキングメモリのコスト」「複数のインターフェース」)

### F2. 数値の短縮表現は禁止

「N+」のような短縮で「N 以上」を表さない  読み手が単位や意味を補完する手間が発生する

| 禁止 | 推奨 |
|---|---|
| 引数 4+ | 引数が 4 個以上 |
| 3+ の変数 | 3 つ以上の変数 |
| 500+ 行 | 500 行以上 |
| 5+ メソッド | 5 メソッド以上 |
| 3+ 箇所 | 3 箇所以上 |

数値 + 単位 + 「以上」/「以下」の形で明示する  単位は文脈に合うものを選ぶ (個 / 件 / 箇所 / 行 / つ 等)

例外: 心理学・UI 用語の正式表記 (Miller の 7±2、Cowan の 4±1、色数 5±2 等) は学術慣例として残す

### F3. 「適用すべきケース」「適用すべきでないケース」は1項目1箇条書きで列挙

複数項目を ` / ` 連結で横並びにせず、1 項目につき 1 箇条書きとして縦に並べる  読み手が項目数と境界を即座に把握でき、追加・削除も差分が見やすい

```md
NG (横並び):
- **適用すべきケース**: フォーム設計 / API パラメータ数 / 関数引数 / 設計議論の論点数

OK (箇条書き):
- **適用すべきケース**:
  - フォーム設計
  - API パラメータ数
  - 関数引数
  - 設計議論の論点数
```

例外: 1 項目内で `string/int` のような連結語 (スペースなし) は分割しない  文として連続している「電話番号・金額・通貨等の業務概念が string/int で流通」のようなまとまりも 1 項目として扱う

例外: **複数の表現が並列していること自体が 1 つのケース** となる場合は分割せず 1 文で書く  「並立している」「対応関係」を示すケースは 1 項目で扱う

```md
NG (誤分割、各行が独立した条件に見える):
- **適用すべきケース**:
  - 業務側「期間限定セール」
  - コード `campaign_discount`
  - 画面「タイムセール」のように呼び方が並立

OK (並立そのものが 1 ケース):
- **適用すべきケース**:
  - 同じ概念に対し、業務側では「期間限定セール」、コードでは `campaign_discount`、画面では「タイムセール」のように呼称がバラバラに並立している
```

### F4. 見出しは「適用すべきでないケース」とだけ書く

`**適用すべきでないケース / 反例**` のような併記をしない  「適用すべきでないケース」と「反例」は同じ概念なので冗長

| 禁止 | 推奨 |
|---|---|
| **適用すべきでないケース / 反例** | **適用すべきでないケース** |

### F5. 「使うとき」の例文に「」を付けない

セリフ調にせず、書き言葉として直接記述する  「セリフを再現するための引用」ではなく「4 点セット表現の例」なので鉤括弧は不要

```md
NG: - 「**SRP** の観点から、{現状 X} を {提案 Y} にすると、{性質 Z} が改善される」
OK: - **SRP** の観点から、{現状 X} を {提案 Y} にすると、{性質 Z} が改善される
```

### F6. 文体は「だ・である調」で統一

辞書として読まれる文書なので、ですます調を混ぜず「だ・である調」に統一する  口語表現 (〜だね / 〜ですよ / 〜しましょう 等) も避ける

| 禁止 | 推奨 |
|---|---|
| 〜が出ています | 〜が出ている |
| 〜なります | 〜なる |
| 〜可能性があります | 〜可能性がある |
| 〜不要です | 〜不要 |
| 〜十分です | 〜十分 |
| 〜たいです | 〜たい |
| 〜しましょう | 〜する |
| 〜しませんか | 〜検討すべき |
| 〜できません | 〜できない |

例外: 「止めるノイズ発言」のように **実際の発言サンプル** として口語表現を引用する場合はそのまま残す (それが意図的な引用であることが文脈で明確な場合に限る)

### F7. 特定プロジェクト・業種に紐づく固有名詞は使わない

特定プロジェクトや特定業界の固有名詞を例文に使わない  読み手のコンテキストに依存して理解が止まる

| 禁止 (PJ・業種固有) | 推奨 (一般化) |
|---|---|
| 外部 API (駅ちか) | 外部 API (決済 API) / 外部 API (在庫 API) |
| 指名料無料 / フリー詳細 / 本指名 | 期間限定セール / 会員割引 / 通常価格 |
| 予約 (特定 PJ 色) | 注文 (EC 系で統一) |

題材は本リポジトリ全体で EC 注文系 (Order / Customer / Payment / Shipping / Notification) に揃える  業務概念の例も金額・通貨・住所・配送先など、業種を選ばない汎用ドメインから選ぶ

### F8. 見出しの表記揺れを許さない

同じ意味の見出しは同じ表記で揃える  揺れがあると grep や目視で識別コストが上がる

| 禁止 | 推奨 |
|---|---|
| **代表的なトレードオフ** | **トレードオフ** |

エントリの標準見出し順 (上から下):

1. `### {タイトル}`
2. `- **読み**:` (必要なら)
3. `- **定義**:`
4. `- **使うとき**:`
5. `- **適用すべきケース**:` (1 項目 1 箇条書き、F3)
6. `- **適用すべきでないケース**:` (1 項目 1 箇条書き、F4)
7. `- **トレードオフ**:`
8. `- **コード例**:` (該当する場合)
9. `- **関連**:`

例外: アンチパターン (Anemic Domain Model 等) は **適用すべきでない反論** 等の特殊見出しを持ってよい  アンチパターン専用の例外フォーマットとして扱う

### F9. 「それっぽいが意味不明」な曖昧表現を使わない

抽象語・主観語で「それっぽく」書くと、読み手 (人間 / AI 双方) が **何を指すか確定できず** 議論や指摘が空中戦化する  「具体・客観・検証可能」な表現に置換する

#### 検出パターン

定期的に以下のキーワードを grep し、文脈で曖昧なら修正する:

| カテゴリ | 検出キーワード例 |
|---|---|
| **対象が不明** | 「構成図」「全体像」「設計思想」「ベストプラクティス」 |
| **基準が不明** | 「合意済み」「合意の上で」「正常な」「適切な」「うまく」「ちょうどいい」「程よく」「健全な」 |
| **主観的修飾** | 「明らかな」「自明な」「常識的な」「本来の」「真の」 |
| **空虚な評価語** | 「綺麗な」「美しい」「クリーンな」「読みやすい」(語彙集の外で単独使用時) |

#### 置換指針

| 軸 | Before (曖昧) | After (具体) |
|---|---|---|
| 抽象 → 具体物 | 「**現コードベースの構成図**に戻る」 | 「**現コードベースのレイヤ構成と依存方向**に戻る」 |
| 主体不明 → 主体明示 | 「**合意済みパターン**を確認」 | 「**現コードベースで採用しているアーキテクチャパターン**を確認」 |
| 主観基準 → 客観基準 | 「**正常な継承**」 | 「**親子の契約が成立している継承**」(LSP の本質を直接表現) |
| 評価語 → 測れる性質 | 「**綺麗になる**」 | 「**変更容易性** が改善される」「**テスト容易性** が改善される」(F6 共通ルール 1 と一貫) |

#### 例外 (残してよい)

- **「コードの臭い」「戦略的設計 / 戦術的設計」** 等の **公式用語** (Fowler / Evans 原著の定訳)
- **「クリーン アーキテクチャ」「クリーン コード」** 等の **固有名詞**
- **数値併記の閾値** (「20〜30 行が目安」「3 箇所以上を目安」) — 数値が具体性を担保
- **引用 / 排除対象として明示しているもの** (「『なんとなく』を排除するための語彙集」のような自己言及)

### F10. AI 生成文章特有の冗長表現を使わない

LLM は「読み手の理解を補助しすぎる」傾向を持つ  「親切すぎ」「網羅しすぎ」「念押ししすぎ」の表現を意識して削る

#### 検出パターン

定期的に以下のキーワードを grep し、文脈で冗長なら削る:

| AI 感タイプ | 検出キーワード | 理由 |
|---|---|---|
| **過剰な再総括** | `つまり` / `すなわち` / `言い換えると` / `要するに` / `結果として` / `結論として` | 既に言ったことを別表現で念押し  読み手の理解を保証したがる癖 |
| **メタ説明の過剰** | `(〜の核)` / `(〜のため)` / `(〜の役割)` / `(〜という意味)` | 概念に効果説明を併記して「重要さ」を印象付けたい癖 |
| **網羅的列挙** | 「A / B / C / D / E / F の統一フォーマット」のような完全列挙 | 機能を漏れなく書いて「全部書いた感」を出したい癖 |
| **空虚な前置き** | `本リポジトリ` / `本ファイル` / `本セクション` / `具体的には` / `実際〜` | 文脈で自明な情報を律儀に明示する癖 |
| **三題噺の癖** | 「A + B + C の組み合わせ」のような美しい 3 つ並び | 3 つに揃えて整理したがる癖 |
| **同じ語の連発・抽象修飾過剰** | 同じ単語が 1 文中に 3 回以上 / 抽象名詞 (`〜性` / `〜化`) の濫用 | 抽象名詞で繕って深さを出したつもりになる癖 |
| **目的と背景の混在** | 「目的」セクションに思想 (Why) が混じる | What と Why をひとまとめにする癖 |
| **「結論+ため」(後置理由) 構造** | 文末が `〜ため` / `〜のため` で終わる長文 | 英語の `to V` 構文を直訳した形 |
| **体言止めと動詞終止形の意味のない混在** | 同じ箇条書きに「〜する」「〜の改善」が混在 / 文末が「〜化」「〜性」「〜さ」「〜の整理」 | リズム取りだけの体言止めをやってしまう癖 |
| **比喩・擬人化・感覚表現** | 擬人化 (神クラス / 何でも知ってる) / 比喩 (腐敗防止 / 貧血) / 感覚表現 (気持ち悪い / 綺麗) | 印象的な言葉で説明したつもりになる癖  初見で具体内容が伝わらない |
| **変更耐性のない数字明示** | 見出し / タイトルに「2 通りの」「4 種の」「6 パターンの」のような具体的な数字 | 網羅性を出したくて入れる癖  内容追加・削減で修正コストが発生 |

#### 修正例

```md
NG (過剰な再総括):
個別セクションで詳細を説明済み

つまり {主軸 1} + {主軸 2} + {主軸 3} の組み合わせ ← 不要な総括

OK:
個別セクションで詳細を説明済み (総括なし)
```

```md
NG (網羅的列挙):
各エントリは「定義 / 使うとき / 適用すべきケース / 適用すべきでないケース / トレードオフ / コード例 / 関連」の統一フォーマット

OK (意図を 1 文で):
各エントリには「適用すべきでないケース」と「トレードオフ」を必ず含める  原則の機械的な振りかざしを防ぐため
```

```md
NG (空虚な前置き):
本ファイルでマッチする症状を抽出
本リポジトリで重視する理由

OK:
マッチする症状を抽出
重視している理由
```

```md
NG (同じ語の連発・抽象修飾過剰):
DDD (業務性担保)
業務ロジック濃いドメインで「業務側との会話の語をコードに 1:1 で乗せる」共通言語  業務エンジニアの軸
(業務 5 連発 + 「業務性担保」のような抽象名詞)

OK (具体ドメイン例 + 目的を明示):
DDD
業務ルールが複雑な領域 (注文 / 在庫 / 会計 / 予約 / 認証等) で、業務側の話す言葉とコードを揃え、仕様変更を局所化するための語彙
```

```md
NG (目的と背景の混在):
## 目的
{What の話}  {Why = すべての精度を語彙が決める}

OK (Why → What を分離):
## 背景
{Why の話}

## 目的
{What の話}
```

```md
NG (結論 + ため、英語的):
各エントリには X を必ず含める  Y を防ぎ、Z で判定する文化に揃えるため

OK (ために + 結論、日本語的):
Y を防ぎ Z で判定する文化に揃えるため、各エントリには X を必ず含める
```

```md
NG (体言止めと動詞終止形の混在):
- A を実装する
- B の改善 ← 体言止め (理由なし)
- C を追加する

OK (統一):
- A を実装する
- B を改善する
- C を追加する
```

```md
NG (擬人化・比喩):
神クラス  何でも知っていて何でもやる巨大クラス  ← 「神」「何でも知ってる」が擬人化
腐敗防止層  外部の語彙を翻訳する隔離層         ← 「腐敗防止」が比喩

OK (具体表現):
複数の責務 (業務ロジック・永続化・通知・整形等) を 1 つのクラスに詰め込んだ結果、行数が大きく変更時の影響範囲が広くなったクラス
外部システム / レガシー DB の語彙やデータ形式を、自分のドメイン語彙に翻訳する隔離層  外部仕様の変更が Domain 層に直接届かないように分離する
```

#### 例外 (残してよい)

- **公式用語の引用** (Fowler の Refactoring カタログ列挙等、機能列挙が本質の場合)
- **意図的な強調** (重要箇所の念押しが文脈で必要な場合)
- **構造化が読みやすさに直結する箇所** (Contents セクション等の目次)
- **比喩そのものが公式用語** (`Code Smell` = 「コードの臭い」、`Anemic Domain Model` = 「貧血ドメインモデル」等)  ただし定義の本文では具体表現に置換する
- **他語彙集で既に定義されている概念**は前提として比喩のまま使ってよい

#### ポジティブ表現要求時の書き換えスタイル

ユーザーから「ポジティブなニュアンスで書いて」と指示があったら、欠如・困りごと表現を能動的・欲求ベース表現に切り替える

| Before (欠如表現) | After (欲求表現) |
|---|---|
| 〜できない / 〜がない | 〜したい / 〜を補強したい |
| 〜が出てこない | 〜を持ちたい |
| 〜暴走する / 〜失敗する | より精度高く〜したい |
| 〜進まない | 〜を前に進めたい |
| 持たない人 | 補強したい人 |

```md
Before (欠如):
- 論理的に言語化する語彙を持たない人
- 相手に伝える言葉が出てこない
- AI が暴走する

After (欲求):
- 論理的に言語化するための語彙を補強したい人
- 相手に伝える具体的な言葉を補強したい
- より精度の高い指示を出したい
```

理由: 「あなたはこれができない」より「あなたはこれをやりたいですよね」の方が読み手が能動的に同意できる  ただし指示無しのデフォルトは中立表現

#### パターン 11. 変更耐性のない数字明示

タイトル・見出し・ファイル名に **具体的な数字** を含めない癖  内容追加 / 削減時に修正コストが発生する

```md
NG (数字明示):
## 人間と AI の 2 通りの使い方     ← 「2 通り」を増減すると見出しも修正
## 4 種のカテゴリ                  ← 「4 種」が増えると修正
## 6 パターンの検出ルール          ← パターン追加で見出しが嘘になる
```

## カテゴリ別 典型ノイズ例

各カテゴリで踏みやすい固有パターン  該当ファイルを書く / FB するときの追加注意

### 設計原則 ([vocab-design-principles.md](vocab-design-principles.md))

- 原則名だけ振りかざして {改善される性質 Z} を言わない (水掛け論化)

### コード品質・リファクタリング ([vocab-code-quality-refactoring.md](vocab-code-quality-refactoring.md))

- 「臭い → リファクタリング手法 → 改善される性質」の 4 点セットを崩さない  臭い名だけだと指摘マウント
- 「今すぐ直すか / 後でいいか」を毎回明示 (Boy Scout Rule / 別 PR / Tech Debt 化)

### アーキテクチャ ([vocab-architecture.md](vocab-architecture.md))

- 「依存方向が逆」と言うときは **何から何への依存** かを必ず明示
- 現コードベースで合意済みの構成図を起点にする  ゼロベース議論は別

### DDD ([vocab-ddd.md](vocab-ddd.md))

- 業務側の語彙と一致しているかを必ず確認  コード上の名前だけで議論しない
- 戦術 (Entity / Value Object / Aggregate) だけで戦略 (Bounded Context / Ubiquitous Language) を無視しない

### テスト容易性 ([vocab-testability.md](vocab-testability.md))

- Test Double (Fake / Stub / Mock / Spy) は厳密に使い分ける  全部「モック」と呼ばない
- 「カバレッジ上げる」だけを目的にしない  設計改善とテスト容易性の同時改善を狙う

### 判断軸・トレードオフ ([vocab-judgment-tradeoffs.md](vocab-judgment-tradeoffs.md))

- 「YAGNI ですよ」「Premature Optimization ですよ」だけで終わらない  「だから今は◯◯にしよう」まで言う
- 判断軸は議論を止める言葉でもある  振りかざせばマウントになる

### 認知負荷 ([vocab-cognitive-load.md](vocab-cognitive-load.md))

- 「認知負荷を下げる」≠「情報を削る」  本質的情報量を保ったまま処理コストを減らす
- 「私には読みづらい」と「多くの読み手にとって読みづらい」は別物
